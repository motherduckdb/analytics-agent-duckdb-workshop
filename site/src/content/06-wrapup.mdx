## The Stack

| Layer | What | Why |
|-------|------|-----|
| **DuckDB** | In-process analytics | No server, just a file |
| **Agentic loop** | LLM + tool calls in a while loop | The model writes SQL, reads errors, self-corrects |
| **Semantic context** | Skill file + `COMMENT ON` | Prevents traps: wrong tables, wrong units, stale data |
| **PydanticAI tools** | `@agent.tool_plain` decorators | Deterministic, auditable results |
| **FastMCP server** | Expose tools over MCP | Any client, any model, zero lock-in |

---

## Key Takeaways

**1. The agentic loop is a solved problem.** Don't build it from scratch — use opencode, Claude Code, or PydanticAI.

**2. Your data breaks it, not the model.** Cents vs euros, stale tables, ambiguous columns. The fix is semantic context, not a better LLM.

**3. Semantic context > semantic layer.** You don't need a separate application. `COMMENT ON` + a skill file get you 90% of the way.

**4. Tools give you determinism.** For board decks and financial reports, `revenue_by_venue(2025)` beats LLM-generated SQL every time.

**5. MCP separates building from distributing.** Build once, serve to any client. The server has no model — just Python functions.
